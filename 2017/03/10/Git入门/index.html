Git入门

发表于 2017-03-10 | 分类于 Git |

Git是目前世界上最先进的分布式版本控制系统。

Git 的基本用法

初始化Git仓库：

git init

添加文件到Git仓库：

git add <file> # 生成一个版本
git commit -m "comments"

查看Git仓库的状态：

git status

查看文件的改动：

git diff <file>

查看提交日志，命令显示从最近到最远的提交：

git log (--pretty=online)

最新提交的版本：

HEAD

上一个版本：

HEAD^

上100个版本：

HEAD~100

在历史版本之间穿梭：

git reset --hard commit_id

Git记录我的每一次命令，方便我们回退到未来的版本：

git reflog

撤回全部的修改：

git checkout -- file # 撤回到最近一次的add和commit的状态

撤回暂存区的修改：

git reset HEAD file # 撤回到最新的版本

删除文件：

# 1
git rm file
git commit -m "delete file"
# 2
git checkout -- file

远程仓库：

添加远程仓库：

# origin 为远程仓库
git remote add origin git@github.com:***/***.git

将本地内容推送到远程仓库：

# 第一次推送加 -u 参数，将本地 master 分支和远程 master 分支关联
git push -u origin master
# 推送
git push origin master

将远程库克隆到本地：

git clone git@github.com:***/***.git

分支管理

HEAD 表示当前分支，我在这个分支上工作，修改和提交都在 HEAD 分支上
master 表示最新的提交，所有的工作都要合并到 master

创建dev分支，并切换到dev分支：

# 1
git checkout -b dev # -b 参数表示切换
# 2
git branch dev # 在此分支上 add 和 commit 都是提交到dev
git checkout dev #

查看当前分支：

git branch

切换分支：

git checkout master

将指定分支合并到 master：

git merge dev
# Fast-forward 快速合并，直接将 master 指向 dev

合并完成后，删除分支：

git branch -d dev

解决冲突：

# 提示 merge conflict，使用 git status 查看，并手动解决冲突
git commit -m "conflict fixed"


# Git 尝试合并，并在文件中标出冲突
<<<<<<< HEAD
Creating a new branch is quick & simple.
=======
Creating a new branch is quick AND simple.
>>>>>>> feature1

查看分支合并情况：

git log --graph --pretty=online --abbrev-commit

>

  • 59bc1cb conflict fixed
    |\
    | * 75a857c AND simple
  • | 400b400 & simple
    |/
  • fec145a branch test

不使用 Fast-forward 模式进行合并:

# --no-ff 使用普通模式进行合并，会产生合并的版本
git merge --no-ff -m "commits" dev

实际开发中的分支策略

master： 稳定版本，不在 master 上进行开发
dev： 开发版本，当稳定时，将其合并到稳定的 master 上
个人分支：每个开发人员都创建自己的分支，不断的提交到 dev 上
Bug分支：创建新的分支修复bug
Feature分支：用于开发新的功能

若手头工作未完成，但是要开辟新的分支：

git stash # 保存工作现场，此时工作区是空的
git stash pop # 恢复工作现场

分支未合并，则需要强制删除

# -D 表示强制删除
git branch -D feature

查看远程库信息：

# 1
git remote
# 2 查看详细信息（fetch抓取，push推送）
git remote -v

GitHub:

fork 别人的仓库到自己的仓库
pull 提交自己的修改
